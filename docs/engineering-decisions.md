# Engineering Decisions & Tradeoffs

This document outlines the core technical decisions made during the architecture of SentinelAI. Every decision was made by balancing system complexity, theoretical correctness, and portfolio value.

---

## 1. Message Queue: Redis (BullMQ) vs. Kafka vs. RabbitMQ

**Decision:** We chose **Redis + BullMQ** as the message broker.

**Why?**
- **Low Latency / High Throughput:** Redis operates entirely in memory, making it exceptionally fast for queuing the high volume of transaction payloads generated by the simulator.
- **Simplicity:** Kafka is the industry standard for event streaming, but managing a ZooKeeper/KRaft cluster and configuring partitioning introduces immense operational overhead for a portfolio project. BullMQ on Redis provides a robust, production-ready queue with retries, rate limiting, and delayed jobs without the weight of Kafka.
- **Portability:** Redis is lightweight to spin up inside `docker-compose`.

## 2. Real-Time Streaming: Server-Sent Events (SSE) vs. WebSockets

**Decision:** We chose **Server-Sent Events (SSE)**.

**Why?**
- **Unidirectional Flow:** The SentinelAI dashboard only needs to *receive* continuous updates from the backend (Live Pipeline, Dashboard KPIs). It does not need to send high-frequency data back to the server. 
- **HTTP Native:** SSE works over standard HTTP/1.1 (and HTTP/2), avoiding the handshake overhead, proxy issues, and custom connection management associated with WebSockets.
- **Built-in Reconnection:** The browser natively attempts to reconnect SSE streams if the connection drops, whereas WebSocket reconnection logic must be manually authored.

## 3. Database: PostgreSQL vs. MongoDB vs. Cassandra

**Decision:** We chose **PostgreSQL** via Prisma ORM.

**Why?**
- **ACID Compliance:** Financial data (fraud logs, transaction records) strictly requires ACID compliance to prevent race conditions during concurrent audits.
- **Relational Integrity:** The schema enforces foreign key constraints between `Transaction`, `User`, `AuditLog`, and `ModelVersion`.
- **Query Flexibility:** Prisma allows us to run complex aggregations on PostgreSQL to power the dashboard analytics (e.g., grouping fraud rates by day) without the complexity of a MapReduce job or column-oriented DB like Cassandra.

## 4. Machine Learning: XGBoost vs. Deep Learning (Neural Networks)

**Decision:** We chose **XGBoost (Extreme Gradient Boosting)**.

**Why?**
- **Tabular Data Superiority:** Fraud detection primarily relies on structured tabular data (amounts, timestamps, categories). Tree-based ensembles like XGBoost empirically outperform deep neural networks on tabular datasets with less tuning.
- **Inference Latency:** XGBoost is heavily optimized for fast CPU inference. When the API Gateway queues a transaction, the FastAPI endpoint can run `.predict_proba()` in milliseconds, which is critical for real-time risk scoring.
- **Native Explainability:** Tree models are natively supported by the `TreeExplainer` in SHAP, allowing exact, fast calculation of feature contributions.

## 5. Explainability: SHAP vs. LIME

**Decision:** We chose **SHAP (SHapley Additive exPlanations)**.

**Why?**
- **Mathematical Consistency:** SHAP is based on cooperative game theory and guarantees local accuracy and consistency. If a feature's true impact increases, its SHAP value will not decrease. LIME does not guarantee this consistency.
- **Global & Local Insight:** We can sum local SHAP values to generate global feature importance for the Model Registry, providing a unified explainability framework.

## 6. Frontend: Next.js App Router vs. React SPA (Vite)

**Decision:** We chose **Next.js (App Router)**.

**Why?**
- **Server Components:** We can offload heavier data-fetching tasks to the server (e.g., initial dashboard loads) while keeping interactive components (like the Explainability Studio) as Client Components.
- **Enterprise Standard:** Next.js provides built-in routing, layout management, and optimized asset loading, matching what top engineering teams use in production.

## 7. Backend Framework: Express vs. NestJS vs. FastAPI

**Decision:** We split the backend into two microservices:
1. **Node.js / Express** for the API Gateway and I/O heavy operations (SSE streaming, Database querying).
2. **Python / FastAPI** for the Machine Learning inference.

**Why?**
- **The Right Tool for the Job:** Node.js excels at asynchronous I/O and connection multiplexing (handling hundreds of SSE clients). However, Python is the unquestioned standard for ML. By splitting them, we prevent CPU-bound XGBoost inference from blocking the Node.js event loop, enabling true enterprise scalability.
